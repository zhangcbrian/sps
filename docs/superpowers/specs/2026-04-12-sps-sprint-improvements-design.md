# SPS 5-Hour Improvement Sprint Design

**Date:** 2026-04-12
**Status:** Approved (autonomous sprint)
**Goal:** Close the most impactful gaps identified in the competitive analysis to differentiate SPS from GitHub Spec Kit and establish the AI agent integration story.

---

## Sprint Priorities (Ranked by Impact)

### 1. AI Agent Instructions Generation (`sps agent`) — HIGH IMPACT

**Why:** The competitive analysis identified AI agent integration as the #1 differentiator opportunity. GitHub Spec Kit works with Copilot/Claude Code/Gemini CLI via freeform Markdown. SPS can generate *structured, actionable* agent instructions from validated specs — something Spec Kit can't do because its specs are unstructured.

**What:** New `sps agent` command that reads all `.sps.yaml` files and generates a `CLAUDE.md` (or similar) file containing:
- All active spec rules as agent-readable instructions
- Given/When/Then as behavioral contracts the agent should follow
- Lineage IDs so the agent knows which test IDs to use
- Cross-cutting references from `touches`

**Core module:** New `packages/core/src/agent.ts` with `generateAgentInstructions(specs, config)` function.
**CLI command:** `sps agent` writes to `CLAUDE.md` (or `--output` flag for custom path).
**Tests:** Unit tests for instruction generation.

### 2. Test Coverage Analysis (`sps coverage`) — HIGH IMPACT

**Why:** SPS has lineage IDs (`REQ-CHECKOUT-COUPON-01`) designed to appear in test `describe` blocks. But there's no tooling to check which specs actually have test coverage. Trace.Space does this; we should too. This is a concrete, measurable value proposition: "SPS tells you which business rules have no tests."

**What:** New `sps coverage` command that:
- Scans all test files (`**/*.test.{ts,js,tsx,jsx}`, `**/*.spec.{ts,js}`, `**/*_test.{go,py}`) for lineage ID patterns (`REQ-xxx-xxx-xx`)
- Cross-references against all rules in `.sps.yaml` files
- Reports covered vs. uncovered rules
- Exits with code 1 if coverage gaps exist (CI-friendly)

**Core module:** New `packages/core/src/coverage.ts` with `analyzeCoverage(specs, repoRoot)` function.
**CLI command:** `sps coverage` with `--strict` flag for CI gating.
**Tests:** Unit tests with temp dirs containing mock test files.

### 3. Principles File (`.sps/principles.yaml`) — MEDIUM IMPACT

**Why:** GitHub Spec Kit has a "constitution" concept — org-wide non-negotiable principles. This is a good idea we should adopt. Principles get included in agent instructions and in the portal.

**What:**
- New optional `.sps/principles.yaml` file with a list of team principles
- Loaded by `loadConfig` (or a new `loadPrinciples` function)
- Included in `sps agent` output
- Shown in portal dashboard
- Schema: simple list of `{ id, title, description }` objects

### 4. CLAUDE.md for SPS Itself — MEDIUM IMPACT

**Why:** We should dogfood our own tool. A CLAUDE.md for the SPS repo helps any agent working on the codebase.

**What:** Write a CLAUDE.md covering:
- Project structure (3 packages)
- How to run tests
- Key conventions (SPS schema, file naming)
- Current branch state

### 5. Improved `sps status` Output — LOW IMPACT

**Why:** Status command is basic. With the manifest, we can show category breakdown, touches graph, and drift warnings.

**What:** Update `sps status` to:
- Show category breakdown (Business: 12, Engineering: 5, Security: 3)
- Show cross-cutting references
- Show drift warnings if manifest exists
- Support `--json` flag for programmatic consumption

## Design Decisions

### Agent Instructions Format

The generated CLAUDE.md section follows this structure:

```markdown
# SPS Spec Rules

> Auto-generated by `sps agent`. Do not edit manually.
> Run `sps agent` to regenerate after spec changes.

## Principles

- **[principle-id]** Title — Description

## Active Rules

### checkout/coupons — Discount Codes

- **REQ-CHECKOUT-COUPON-01** Customers can use a percentage discount code
  - Given: A customer has a cart totaling $100 and enters coupon SAVE20 (20% off, not expired)
  - When: The customer applies the coupon at checkout
  - Then: Cart total drops to $80. The $20 discount appears as a line item.
  - Tests: Link test describe blocks with `[REQ-CHECKOUT-COUPON-01]`

### billing — Billing & Invoicing (touches: checkout)

...
```

### Coverage Analysis

The coverage scanner uses a regex pattern to find lineage IDs in test files:
```
/REQ-[A-Z]+-[A-Z]+-\d{2}/g
```

It produces a report:
```
SPS Test Coverage Report
========================
Total rules:     16
Covered:         12 (75%)
Uncovered:       4

Uncovered rules:
  REQ-CHECKOUT-COUPON-03  src/checkout/coupons/coupons.sps.yaml
  REQ-BIL-INV-02          src/billing/invoices.sps.yaml
  ...
```

### Principles Schema

```yaml
# .sps/principles.yaml
principles:
  - id: no-silent-failures
    title: "No silent failures"
    description: "Every error path must log, alert, or return an error. Never swallow exceptions."
  - id: money-in-cents
    title: "Money in cents"
    description: "All monetary values are integers in cents. Never use floating point for money."
```
